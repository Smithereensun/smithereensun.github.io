<h1 style="text-align: center">网关</h1>
<p>　　负责路由转发+过滤器；他是系统的唯一对外的入口，介于客户端和服务器之间的中间层，处理非业务功能，提供路由请求、鉴权、监控、缓存、限流等功能</p>
<h1 style="text-align: center">服务注册发现</h1>
<p>　　<span style="color: rgba(255, 0, 0, 1)"><strong>调用和被调用方信息维护</strong></span>；服务启动的时候，都注册到注册中心里，这样的话别人调用的时候，就知道有哪些ip地址和端口号了</p>
<h1 style="text-align: center">配置中心</h1>
<p>　　管理配置，动态更新</p>
<h1 style="text-align: center">链路追踪</h1>
<p>　　分析调用链路耗时；例如：下单、查库存、减库存、付款、下单完成</p>
<h1 style="text-align: center">负载均衡器</h1>
<p>　　分发负载；例如：nginx</p>
<h1 style="text-align: center">熔断</h1>
<p>　　保护自己和被调用方，类似于家里的保险丝，为了防止整个系统故障</p>
<h1 style="text-align: center">降级</h1>
<p>　　服务降级是当服务器压力剧增的情况下，根据当前业务情况及流量对一些服务和页面有策略的降级，以此释放服务器资源以保证核心任务的正常运行。</p>
<h1 style="text-align: center">熔断和降级区别</h1>
<h2>相同点</h2>
<ol>
<li>从可用性和可靠性触发，为了防止系统崩溃</li>
<li>最终让用户体验到的是某些功能暂时不能用</li>
</ol>
<h2>不同点</h2>
<ol>
<li>服务熔断一般是下游服务故障导致的</li>
<li>服务降级一般是从整个系统负荷考虑，由调用方控制</li>
</ol>
<p>　　抛弃一些非核心的接口和数据，例如：双11，支付宝将查询当月账单功能临时降级等等</p>
<h1 style="text-align: center">什么是微服务的注册中心</h1>
<p>　　服务管理，核心是有个服务注册表，心跳机制动态维护</p>
<h2>服务提供者</h2>
<p>　　启动的时候向注册中心上报自己的网络信息</p>
<h2>服务消费者</h2>
<p>　　启动的时候向注册中心上报自己的网站信息，拉取提供者的相关网络信息</p>
<h2>为什么要用注册中心</h2>
<p>　　微服务应用和机器越来越多，调用方需要知道接口的网络地址，如果靠配置文件的方式去控制网络地址，对于动态新增机器，维护起来很麻烦</p>
<h2>主流的注册中心</h2>
<ol>
<li>zookeeper</li>
<li>eureka</li>
<li>consul</li>
<li>etcd</li>
<li>nacos</li>
<li>等</li>
</ol>
<h1 style="text-align: center">分布式应用CAP理论知识</h1>
<h2>定理</h2>
<p>　　指的是在一个分布式系统中，Consistency(<strong><span style="color: rgba(255, 0, 0, 1)">一致性</span></strong>)、Availability(<span style="color: rgba(255, 0, 0, 1)"><strong>可用性</strong></span>)、Partition tolerance(<span style="color: rgba(255, 0, 0, 1)"><strong>分区容错性</strong></span>)，<span style="color: rgba(255, 0, 0, 1)"><strong>三者不可同时获得</strong></span>。<span style="color: rgba(255, 0, 0, 1)"><strong>CAP理论</strong></span>就是说<span style="color: rgba(255, 0, 0, 1)"><strong>在分布式存储系统中</strong></span>，<span style="color: rgba(255, 0, 0, 1)"><strong>最多只能实现上面的两点</strong></span>。而由于当前的网络硬件肯定会出现延迟丢包等问题，所以<span style="color: rgba(255, 0, 0, 1)"><strong>分区容错性</strong></span>是我们<span style="color: rgba(255, 0, 0, 1)"><strong>必须需要实现的</strong></span>，所以我们只能在<span style="color: rgba(255, 0, 0, 1)"><strong>一致性和可用性之间进行权衡</strong></span>。</p>
<h2>一致性(<span style="color: rgba(255, 0, 0, 1)">C</span>)</h2>
<p>　　在分布式系统中的所有数据备份，在同一时刻是否同样的值(所有节点在同一时间的数据完全一致，越多数据同步越耗时)。</p>
<h2>可用性(<span style="color: rgba(255, 0, 0, 1)">A</span>)</h2>
<p>　　负载过大后，集群整体是否还能响应客户端的读写请求(服务一直可用，而且是正常响应时间)。</p>
<h2>分区容错性(<span style="color: rgba(255, 0, 0, 1)">P</span>)</h2>
<p>　　分区容忍性，就是高可用性，一个节点崩了，并不影响其他的节点。</p>
<h2>CA(<span style="color: rgba(255, 0, 0, 1)">一致性、可用性</span>)</h2>
<p>　　数据同步(<strong><span style="color: rgba(255, 0, 0, 1)">C</span></strong>)需要时间，也要正常的时间内响应(<strong><span style="color: rgba(255, 0, 0, 1)">A</span></strong>)，那么机器数量就要少，所以<span style="color: rgba(255, 0, 0, 1)"><strong>P</strong></span>就不满足。</p>
<h2>CP(<span style="color: rgba(255, 0, 0, 1)">一致</span><span style="color: rgba(255, 0, 0, 1)">性、分区容错性</span>)</h2>
<p>　　数据同步(<span style="color: rgba(255, 0, 0, 1)"><strong>C</strong></span>)需要时间，机器数量也多(<span style="color: rgba(255, 0, 0, 1)"><strong>P</strong></span>)，但是同步数据需要时间，所以不能再正常时间内响应，所以<span style="color: rgba(255, 0, 0, 1)"><strong>A</strong></span>就不满足。</p>
<h2>AP(<span style="color: rgba(255, 0, 0, 1)">可用性、分区容错性</span>)</h2>
<p>　　机器数量也多(<span style="color: rgba(255, 0, 0, 1)"><strong>P</strong></span>)，正常的时间内响应(<span style="color: rgba(255, 0, 0, 1)"><strong>A</strong></span>)，那么数据就不能及时同步其他节点，所以<span style="color: rgba(255, 0, 0, 1)"><strong>C</strong></span>不满足。</p>
<h2>注册中心选择</h2>
<p>　　Zookeeper：CP原则，保证了一致性，集群搭建的时候，某个节点失效，则会进行选举新的leader，或者半数以上节点不可用，则无法提供服务，因此可用性没法满足</p>
<p>　　Eureka：AP原则，无主从节点，一个节点挂了，自动切换其他节点可以使用，去中心化</p>
<h1 style="text-align: center">服务间调用方式</h1>
<h2>RPC</h2>
<p>　　远程过程调用，像调用本地服务(方法)一样调用服务器的服务，支持同步、异步调用，客户端和服务器之间建立TCP连接，可以一次建立一个，也可以多个调用复用一次连接</p>
<h2>REST(<span style="color: rgba(255, 0, 0, 1)">Http</span>)</h2>
<p>　　http请求，支持多种协议和功能，开发方便成本低</p>
